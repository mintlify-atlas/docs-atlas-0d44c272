Mercury is an AI agent that runs on your machine, connects to LLMs, and carries out multi-step tasks through 31 built-in tools — all while asking permission before it acts. Unlike typical AI wrappers that execute silently, Mercury treats your filesystem, shell, and data as things worth protecting. It is built around a strict permission system, a configurable personality, and a token budget, and it runs 24/7 through the CLI or Telegram.

## What makes Mercury different

Most AI agents can read files, run commands, and fetch URLs. Mercury does all of that — and then asks first.

## Architecture overview

Mercury is built on TypeScript and Node.js 20+, using the Vercel AI SDK v4 for its agentic loop. It is structured around four major layers:

<CardGroup cols={2}>
  <Card title="Core agent loop" icon="arrows-spin">
    `src/core/agent.ts` runs a multi-step `generateText()` loop with up to 10 steps per request. The LLM decides whether to respond or call a tool — and the loop continues until a final text response is produced.
  </Card>
  <Card title="Capabilities and permissions" icon="lock">
    `src/capabilities/` registers all 31 tools. `src/capabilities/permissions.ts` enforces folder-level scoping and the shell blocklist before any tool executes.
  </Card>
  <Card title="Channels" icon="satellite-dish">
    `src/channels/` provides abstract channel adapters. `cli.ts` handles readline input and inline permission prompts. `telegram.ts` manages the grammY bot, streaming, and file uploads.
  </Card>
  <Card title="Soul and memory" icon="brain">
    `src/soul/identity.ts` loads the soul files that define Mercury's personality and system prompt. `src/memory/store.ts` maintains short-term, episodic, and long-term memory across sessions.
  </Card>
</CardGroup>

### The agentic loop in detail

```
User message → agent loads system prompt (soul + guardrails + persona)
  → generateText({ tools, maxSteps: 10 })
    → LLM decides: text response OR tool call
      → Tool call: permission check → execute → return result to LLM
      → LLM continues (may call more tools, or produce final response)
  → Final response sent to user via channel
```

<Note>
  Mercury's tools are registered in `src/capabilities/registry.ts`. All tool calls pass through the permission manager before execution — denied calls return a denial message to the LLM so it can adjust its approach.
</Note>

## Agent lifecycle

Mercury progresses through a state machine for each interaction:

```
unborn → birthing → onboarding → idle ⇄ thinking → responding → idle
                                                         ↓
                                           idle → sleeping → awakening → idle
```

<Tip>
  The lifecycle state is managed in `src/core/lifecycle.ts`. In daemon mode, Mercury moves between idle and sleeping states according to the heartbeat schedule — waking to send proactive messages or run scheduled tasks.
</Tip>

## Flat-file persistence

Mercury uses no database. All runtime data lives in `~/.mercury/` as YAML and JSON files. This means your configuration, soul files, memory, and permissions are plain text you can read, edit, and back up at any time.
