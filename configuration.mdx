All of Mercury's runtime data lives in `~/.mercury/` — not in your project directory. This keeps configuration separate from your code and makes it easy to use one Mercury installation across multiple projects. The fastest way to change any setting is `mercury doctor`, which runs an interactive reconfiguration wizard.

## Runtime data directory

| Path | Purpose |
|------|---------|
| `~/.mercury/mercury.yaml` | Main config file (providers, channels, budget, identity) |
| `~/.mercury/.env` | API keys and tokens |
| `~/.mercury/soul/*.md` | Agent personality files |
| `~/.mercury/permissions.yaml` | Filesystem and shell permission manifest |
| `~/.mercury/skills/` | Installed skills |
| `~/.mercury/schedules.yaml` | Scheduled tasks (cron jobs) |
| `~/.mercury/token-usage.json` | Daily token usage tracking |
| `~/.mercury/memory/` | Short-term, episodic, and long-term memory |
| `~/.mercury/daemon.pid` | Background process PID (daemon mode) |
| `~/.mercury/daemon.log` | Daemon mode logs |

<Tip>
  You can set the `MERCURY_HOME` environment variable to override the `~/.mercury/` directory location. Useful when running multiple isolated Mercury instances.
</Tip>

## The `mercury.yaml` config file

`~/.mercury/mercury.yaml` is the main configuration file. It is generated by the setup wizard and updated by `mercury doctor`. You can also edit it directly — Mercury reads it on every start.

The file follows the structure of the `MercuryConfig` interface in `src/utils/config.ts`. A typical config looks like:

```yaml
identity:
  name: Mercury
  owner: Alice

providers:
  default: deepseek
  deepseek:
    name: deepseek
    apiKey: sk-xxxxxxxxxxxxxxxx
    baseUrl: https://api.deepseek.com/v1
    model: deepseek-chat
    enabled: true
  openai:
    name: openai
    apiKey: sk-xxxxxxxxxxxxxxxx
    baseUrl: https://api.openai.com/v1
    model: gpt-4o-mini
    enabled: true

channels:
  telegram:
    enabled: true
    botToken: "123456789:AAXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX"
    streaming: true
    admins: []
    members: []
    pending: []

memory:
  shortTermMaxMessages: 20

heartbeat:
  intervalMinutes: 60

tokens:
  dailyBudget: 1000000
```

<Note>
  Settings in `mercury.yaml` take precedence over environment variables from `.env`. Values are deep-merged, so you only need to include the keys you want to override.
</Note>

## API keys (`.env` file)

API keys and sensitive tokens are stored separately in `~/.mercury/.env`. This file is loaded at startup alongside any `.env` in your current project directory.

```bash
# ~/.mercury/.env

# Agent identity
MERCURY_NAME=Mercury
MERCURY_OWNER=Alice

# LLM providers
DEEPSEEK_API_KEY=sk-xxxxxxxxxxxxxxxx
DEEPSEEK_BASE_URL=https://api.deepseek.com/v1
DEEPSEEK_MODEL=deepseek-chat
OPENAI_API_KEY=sk-xxxxxxxxxxxxxxxx
OPENAI_BASE_URL=https://api.openai.com/v1
OPENAI_MODEL=gpt-4o-mini
ANTHROPIC_API_KEY=sk-ant-xxxxxxxxxxxxxxxx
ANTHROPIC_MODEL=claude-sonnet-4-20250514
GROK_API_KEY=xai-xxxxxxxxxxxxxxxx
GROK_BASE_URL=https://api.x.ai/v1
GROK_MODEL=grok-4
OLLAMA_CLOUD_API_KEY=your-api-key
OLLAMA_CLOUD_BASE_URL=https://ollama.com/api
OLLAMA_CLOUD_MODEL=gpt-oss:120b
OLLAMA_LOCAL_BASE_URL=http://127.0.0.1:11434/api
OLLAMA_LOCAL_MODEL=gpt-oss:20b
OLLAMA_LOCAL_ENABLED=false
DEFAULT_PROVIDER=deepseek

# Telegram
TELEGRAM_BOT_TOKEN=123456789:AAXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX

# GitHub (optional)
GITHUB_TOKEN=ghp_xxxxxxxxxxxxxxxx
GITHUB_USERNAME=yourhandle
GITHUB_EMAIL=mercury@cosmicstack.org
GITHUB_DEFAULT_OWNER=yourhandle
GITHUB_DEFAULT_REPO=yourrepo

# Token budget
DAILY_TOKEN_BUDGET=50000

# Heartbeat
HEARTBEAT_INTERVAL_MINUTES=60
```

<Warning>
  Never commit `~/.mercury/.env` to version control. It is stored in your home directory outside of any project repo by design.
</Warning>

## LLM providers

Mercury supports six providers with automatic fallback. It tries the default provider first, then falls back to the next configured one in order. It remembers the last successful provider and starts there on the next request.

| Provider | Default model | API key variable | Notes |
|----------|--------------|------------------|-------|
| **DeepSeek** | `deepseek-chat` | `DEEPSEEK_API_KEY` | Default provider, cost-effective |
| **OpenAI** | `gpt-4o-mini` | `OPENAI_API_KEY` | Supports GPT-4o, o3, and more |
| **Anthropic** | `claude-sonnet-4-20250514` | `ANTHROPIC_API_KEY` | Claude Sonnet, Haiku, Opus |
| **Grok (xAI)** | `grok-4` | `GROK_API_KEY` | OpenAI-compatible endpoint |
| **Ollama Cloud** | `gpt-oss:120b` | `OLLAMA_CLOUD_API_KEY` | Remote Ollama via `ollama.com/api` |
| **Ollama Local** | `gpt-oss:20b` | No key needed | Local Ollama at `127.0.0.1:11434` |

To change the default provider, update `providers.default` in `mercury.yaml` or the `DEFAULT_PROVIDER` environment variable. Valid values are `deepseek`, `openai`, `anthropic`, `grok`, `ollamaCloud`, and `ollamaLocal`.

<Accordion title="Custom OpenAI-compatible endpoints">
  The OpenAI provider supports custom base URLs via `OPENAI_BASE_URL`. This lets you point Mercury at any OpenAI-compatible API, such as a local LM Studio server or a third-party proxy:

  ```bash
  OPENAI_BASE_URL=http://localhost:1234/v1
  OPENAI_MODEL=your-model-name
  OPENAI_API_KEY=not-needed  # Some local servers ignore this
  ```
</Accordion>

<Accordion title="Disabling a provider">
  Set the `enabled` flag to `false` in `mercury.yaml`, or add the corresponding environment variable:

  ```bash
  OPENAI_ENABLED=false
  ANTHROPIC_ENABLED=false
  ```

  Disabled providers are skipped during fallback.
</Accordion>

## Soul files

Mercury's personality is defined by four markdown files in `~/.mercury/soul/`. You own these files and can edit them freely — Mercury loads them into its system prompt on every request.

| File | Human analogy | Purpose |
|------|---------------|---------|
| `soul.md` | Heart | Core values, principles, and identity |
| `persona.md` | Face | Communication style and outward personality |
| `taste.md` | Palate | Aesthetic preferences and decision-making heuristics |
| `heartbeat.md` | Breathing | Proactive behavior, what Mercury does when idle |

The soul files are seeded with defaults during first-time setup. Edit them like any markdown file to shape how Mercury thinks, speaks, and acts.

<Tip>
  The soul system prompt (soul + guardrails + persona) costs approximately 500 tokens per request. Keeping soul files concise helps preserve your token budget.
</Tip>

## Token budget

Mercury enforces a daily token budget to prevent runaway costs. The default is 1,000,000 tokens per day — set a lower value via `DAILY_TOKEN_BUDGET` if you want tighter cost control.

| Setting | Variable | Default | Description |
|---------|----------|---------|-------------|
| Daily budget | `DAILY_TOKEN_BUDGET` | `1000000` | Total tokens allowed per day |
| Heartbeat interval | `HEARTBEAT_INTERVAL_MINUTES` | `60` | Minutes between proactive heartbeat checks |

When usage reaches 70% of the daily budget, Mercury automatically shortens its responses. At 100%, it stops accepting new requests until the budget resets or is overridden.

Manage the budget during a conversation with in-chat commands:

```
/budget              # Show current usage and limit
/budget reset        # Reset usage counter to zero
/budget override     # Override the limit for the current request
/budget set 100000   # Change the daily limit to 100,000 tokens
```

Usage is tracked in `~/.mercury/token-usage.json` and resets automatically at midnight local time.

## Permissions manifest

`~/.mercury/permissions.yaml` stores the filesystem scopes and shell approvals you have granted. Mercury reads this file on every tool call — you can edit it directly at any time, and changes take effect immediately without a restart.

When Mercury needs a permission it doesn't have, it asks inline:

```
⚠ Mercury needs write access to ~/projects/myapp. Allow? (y/n/always):
> always
[Scope saved to ~/.mercury/permissions.yaml]
```

- **`y`** — grant once for this request only
- **`always`** — grant permanently (saved to `permissions.yaml`)
- **`n`** — deny; Mercury adjusts its approach

### Shell permission tiers

| Tier | Examples | Behavior |
|------|----------|----------|
| **Blocked** (never run) | `sudo *`, `rm -rf /`, `mkfs`, `dd if=`, fork bombs, `shutdown`, `reboot` | Hard-blocked; cannot be approved |
| **Auto-approved** | `ls`, `cat`, `pwd`, `git status`, `git diff`, `node`, `npm run` | No prompt required |
| **Needs approval** | `npm publish`, `git push`, `docker`, `rm -r`, `chmod`, `curl \| sh` | Mercury asks before running |

## Telegram configuration

To enable Telegram, set `TELEGRAM_BOT_TOKEN` in `~/.mercury/.env` (or provide it during `mercury doctor`). Create a bot via [@BotFather](https://t.me/BotFather) on Telegram to get a token.

```bash
TELEGRAM_BOT_TOKEN=123456789:AAXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX
```

Mercury uses an organization access model. The first user to connect becomes an admin; additional users request access and must be approved by an admin.

**First-time Telegram setup:**

1. Start your bot — send `/start` in Telegram
2. You'll receive a pairing code
3. Approve yourself from the CLI: `mercury telegram approve <code>`
4. You are now the first admin

**Managing access from the CLI:**

```bash
mercury telegram list                    # List all approved and pending users
mercury telegram approve <code|id>      # Approve a pairing code or pending request
mercury telegram reject <id>            # Reject a pending access request
mercury telegram remove <id>            # Remove an approved user
mercury telegram promote <id>           # Promote a member to admin
mercury telegram demote <id>            # Demote an admin to member
mercury telegram unpair                 # Clear all access and start fresh
```

<Note>
  Telegram only accepts private chats — group messages are always ignored. Streaming can be toggled with `/stream` or `/stream off` during a conversation.
</Note>

## GitHub integration

Mercury supports optional GitHub integration so it can create pull requests, manage issues, review code, and co-author commits on your behalf. Configure it during `mercury doctor` or by setting environment variables.

| Variable | Description |
|----------|-------------|
| `GITHUB_TOKEN` | Personal Access Token with `repo` scope (or fine-grained: Contents + Pull requests + Issues) |
| `GITHUB_USERNAME` | Your GitHub username |
| `GITHUB_EMAIL` | Email for commit attribution (defaults to `mercury@cosmicstack.org`) |
| `GITHUB_DEFAULT_OWNER` | Default repository owner for GitHub tool calls |
| `GITHUB_DEFAULT_REPO` | Default repository name for GitHub tool calls |

When `GITHUB_TOKEN` is set, Mercury registers additional GitHub tools: `create_pr`, `review_pr`, `list_issues`, `create_issue`, and `github_api`. See [Git tools](/reference/tools-git#github-tools) for full parameter documentation.

<Tip>
  Use a fine-grained PAT for minimum required permissions: Contents (read/write), Pull requests (read/write), Issues (read/write). Create one at [github.com/settings/personal-access-tokens/new](https://github.com/settings/personal-access-tokens/new).
</Tip>

## Reconfigure with `mercury doctor`

The `mercury doctor` command is the recommended way to change any setting after first-time setup. It walks through all configuration options and shows current values — press Enter to keep any value, or type a new one.

```bash
mercury doctor
```

To see the current state of your configuration without making changes:

```bash
mercury status
```
