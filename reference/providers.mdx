---
title: "LLM providers: configure and switch AI backends"
description: "Mercury supports DeepSeek, OpenAI, Anthropic, Grok, and Ollama with automatic fallback. Configure multiple providers and Mercury tries them in priority order."
---

Mercury connects to six LLM providers and uses them with automatic fallback. You configure which providers are active during the setup wizard or by running `mercury doctor` at any time. API keys are stored in `~/.mercury/.env` and loaded alongside your project's `.env` on startup.

## Provider fallback

When Mercury makes an LLM request, it starts with the default provider — DeepSeek by default, or whichever provider you set as default during setup. If that request fails, Mercury automatically tries the next configured provider in order, continuing until one succeeds or all fail.

Mercury also tracks the last successful provider across requests. On the next call, it starts from that provider rather than always restarting from the configured default. This means if one provider has an outage, Mercury shifts to an alternative and stays there until something changes.

The fallback order is determined by the order providers are registered in `ProviderRegistry`: DeepSeek → OpenAI → Anthropic → Grok → Ollama Cloud → Ollama Local. Only providers with a configured API key (or, for Ollama Local, a valid base URL) are registered.

<Tip>
  Configure at least two providers so Mercury has a fallback if one is unavailable. DeepSeek and OpenAI together make a cost-effective primary/fallback pair.
</Tip>

    | Setting | Value |
    |---------|-------|
    | Default model | `gpt-oss:20b` |
    | API key env var | None required |
    | Base URL | `http://127.0.0.1:11434/api` |
    | Architecture | OpenAI-compatible |
    | Enabled by default | No |

    ```bash
    OLLAMA_LOCAL_ENABLED=true

    OLLAMA_LOCAL_BASE_URL=http://127.0.0.1:11434/api
    ```

      Ollama Local is disabled by default. Enable it explicitly via `OLLAMA_LOCAL_ENABLED=true` or during `mercury doctor`.

    ```bash
    ollama pull llama3.2:3b
    ```

## Configure providers

Providers are configured during the initial setup wizard. To change keys, models, or which providers are active at any time, run:

```bash
mercury doctor
```

`mercury doctor` shows your current configuration with masked keys and lets you update each provider. Press Enter to keep an existing value.

During setup, Mercury fetches the available model list from each provider's API — so you can select from models your account actually has access to, rather than typing a model name manually.

## Change the default provider

The default provider is set with the `DEFAULT_PROVIDER` environment variable or chosen during setup. Valid values are `deepseek`, `openai`, `anthropic`, `grok`, `ollamaCloud`, and `ollamaLocal`.

```bash
# ~/.mercury/.env
DEFAULT_PROVIDER=openai
```

After changing this, restart Mercury for the new default to take effect.

## API key storage

All API keys are stored in `~/.mercury/.env`. Mercury loads this file automatically on startup alongside any `.env` in your working directory.

```bash
# ~/.mercury/.env
DEEPSEEK_API_KEY=sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
OPENAI_API_KEY=sk-proj-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
ANTHROPIC_API_KEY=sk-ant-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
GROK_API_KEY=xai-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
OLLAMA_CLOUD_API_KEY=xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
DEFAULT_PROVIDER=deepseek
```

You can also set these as regular shell environment variables — Mercury reads `process.env` so any value present in the environment at startup will be picked up.

<Warning>
  Do not commit `~/.mercury/.env` to version control. It contains your API keys.
</Warning>

## API key format validation

Mercury validates key format at setup time and surfaces an error immediately if the format doesn't match what the provider expects.

| Provider | Required format |
|----------|----------------|
| OpenAI | Starts with `sk-`, `sk-proj-`, or `sk-svcacct-` |
| Anthropic | Starts with `sk-ant-` |
| DeepSeek | Starts with `sk-` |
| Grok | No enforced prefix |
| Ollama Cloud | No enforced prefix |
| Ollama Local | No key required |

Format validation runs at configuration time only. At runtime, Mercury relies on provider API responses to detect authentication failures and trigger fallback.

## OpenAI-compatible architecture

Most Mercury providers — DeepSeek, OpenAI, Grok, Ollama Cloud, and Ollama Local — share the same `OpenAICompatProvider` implementation, which accepts a `baseUrl` alongside the API key. This means you can point any of these providers at a custom endpoint by overriding the base URL environment variable:

```bash
# Use a local OpenAI-compatible proxy for OpenAI
OPENAI_BASE_URL=http://localhost:8080/v1
OPENAI_API_KEY=not-needed

# Use a self-hosted Grok-compatible endpoint
GROK_BASE_URL=https://my-xai-proxy.example.com/v1
```

Anthropic uses a separate native SDK integration (`AnthropicProvider`) and does not share this base URL mechanism, though `ANTHROPIC_BASE_URL` is still read from the environment for future use.
